# Git 커밋 & 푸시

변경사항을 사용자의 커밋 컨벤션에 따라 스테이징, 커밋, 푸시하는 스킬입니다. 사용자는 **간결한** 커밋 메시지와 고정된 커밋 타입을 원합니다.

## 커밋 컨벤션

### 허용 타입 (아래 다섯 종류만)

| 타입 | 사용 대상 |
|------|----------|
| `feat` | 새로운 기능 또는 동작 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 변경만 있는 경우 |
| `chore` | 빌드, 설정, 의존성, 도구, 비코드성 유지보수 |
| `test` | 테스트 추가 또는 수정 |

위 목록 외 타입은 사용하지 않습니다 (`refactor`, `style`, `perf`, `ci` 등 금지). 변경이 다섯 타입 중 하나로 명확히 매핑되지 않으면 추측하지 말고 사용자에게 어떤 타입을 쓸지 물어봅니다.

### 메시지 형식

```
<type>: <subject>
```

규칙:
- **간결하게.** subject는 짧은 요약이며, 마침표로 끝나는 문장이 아닙니다. 설명문이 아닌 압축된 구문을 지향합니다.
- 커밋 하나당 하나의 논리적 변경. 스테이징된 변경이 여러 타입에 걸쳐 있으면 커밋 분리를 제안합니다.
- subject 언어는 저장소 컨벤션을 따릅니다. 기존 히스토리가 한국어면 한국어로, 영어면 영어로 작성합니다. 불명확하면 한 번 물어봅니다.
- 사용자가 명시적으로 요청하지 않는 한 본문(body)은 작성하지 않습니다.

예시:
```
feat: 알림 엔티티 추가
fix: 스케줄러 중복 실행 방지
docs: README 설치 절차 보강
chore: 의존성 버전 업데이트
test: 알림 발송 단위 테스트 추가
```

## 작업 흐름

아래 순서를 따릅니다. 검토 단계를 건너뛰지 않습니다.

### 1. 현재 상태 확인

다음을 순서대로 실행합니다:
```bash
git status
git diff --stat
git diff          # 스테이징 + 미스테이징 변경 내용 파악
```

커밋할 변경이 없으면 중단하고 사용자에게 알립니다.

### 2. 타입 및 메시지 결정

- 실제 diff를 읽고 올바른 타입을 결정합니다. 파일명만으로 타입을 추론하지 않습니다.
- diff에 서로 무관한 변경이 섞여 있으면(예: 기능 + 무관한 설정 변경) 이를 알리고 하나의 혼합 커밋 대신 분리를 제안합니다.
- 위 형식에 맞춰 간결한 메시지를 작성합니다.

### 3. 커밋 전 확인

스테이징 대상 파일과 제안하는 커밋 메시지를 사용자에게 보여준 뒤 **승인을 기다립니다.** 명시적 확인 없이 커밋하지 않습니다.

다음과 같이 제시합니다:
```
스테이징 대상: <files>
커밋 메시지: <type>: <subject>
```

### 4. 스테이징 및 커밋

승인 후:
```bash
git add <paths>          # 의도한 파일만 스테이징, 사용자가 요청하지 않는 한 무분별한 `git add -A` 지양
git commit -m "<type>: <subject>"
```

### 5. 푸시

푸시는 변경을 원격에 게시하는 작업이므로 **항상 확인 후 진행합니다.** 먼저 브랜치와 원격을 명시합니다:
```bash
git rev-parse --abbrev-ref HEAD      # 현재 브랜치
git remote -v
```
사용자 확인 후:
```bash
git push origin <branch>
```

브랜치에 업스트림이 설정되지 않은 경우:
```bash
git push -u origin <branch>
```

## 하지 말아야 할 것

- `feat`, `fix`, `docs`, `chore`, `test` 외 커밋 타입 사용 금지.
- 명시적 요청 없이 장황한 여러 줄 메시지나 본문 작성 금지.
- 무엇이 일어날지 보여주고 확인받지 않은 채로 커밋 또는 푸시 금지.
- 사용자가 명시적으로 요청하고 결과를 이해하지 않는 한 force-push(`git push -f`), 히스토리 재작성, 게시된 커밋 amend 금지.
- 작업 트리에 무관한 변경이 있을 때 무분별한 `git add -A` / `git add .` 금지 — 의도적으로 스테이징.
- 변경이 모호할 때 커밋 타입 추측 금지 — 물어볼 것.

## 응답 스타일

- 사용자가 한국어로 작성하면 기본적으로 한국어로 응답합니다. 사용자 언어에 맞춥니다.
- 간결하게. 명령어와 관련 출력만 보여주고 모든 단계를 일일이 서술하지 않습니다.
- diff 요약과 제안 메시지를 명확히 드러내 사용자가 한눈에 승인할 수 있게 합니다.
