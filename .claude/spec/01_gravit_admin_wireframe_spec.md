## 1. 서비스 배경
 
### Gravit 서비스
- **CS(Computer Science) 학습 서비스**
- 사용자가 챕터 → 유닛 → 레슨 → 문제(객관식/주관식)를 풀면서 학습
- 학습 컨텐츠는 별도 파이프라인(LLM 기반)으로 자동 생성 → 스테이징 테이블 적재 → 운영자 검수 후 → prod 테이블로 promote
### 백오피스의 목적
- 운영자(내부 직원, 수십 명 이내)가 사용
- 주요 업무:
  - 유저 관리 (상태/역할 변경)
  - 신고 처리 (유저가 문제에 신고를 남김)
  - 학습 컨텐츠 관리 (운영 중인 컨텐츠 수정, 챕터 풀이 현황 확인)
  - 스테이징 컨텐츠 검수 및 promote (가장 빈번한 작업)
  - 공지 관리

## 2. 프로젝트 개요
 
| 항목 | 결정 사항 |
|---|---|
| **디바이스 / 뷰포트** | 데스크탑 전용, 고정 폭 1280px+ (반응형 미고려) |
| **디자인 시스템** | shadcn/ui 기반 (Radix + Tailwind) |
| **톤 & 무드** | 미니멀 & 정보 중심 (Linear, Vercel Dashboard 계열) |
| **Primary 컬러** | `#7738EE` (Gravit 서비스 핵심 컬러, 포인트 사용 원칙) |
| **컬러 모드** | 라이트 모드 전용 |
| **검색 / 필터 정책** | 혼합 방식 — 유저·신고·스테이징 라벨에만 검색·필터 UI 노출, 챕터/유닛/레슨/공지는 페이지네이션만 |
| **위험 액션 확인** | 단순 confirm 모달 (모든 위험 액션 통일) |
| **네비게이션** | 좌측 사이드바 고정 (240px) |
| **Audit log** | 미표시 (현재 API 미지원) |
| **아이콘 라이브러리** | lucide-react |
 
### 컬러 사용 원칙
- `#7738EE`는 채도가 높은 바이올렛 계열로, **액션 버튼·활성 상태·강조 요소에만 집중적으로 사용**
- 텍스트나 큰 면적에는 사용하지 않음
- 위험 액션(destructive)에는 빨간 계열, 일반 액션에는 Primary 컬러 적용

#### 사양
 
- 화면 배경: `#FAFAFA` (옅은 회색)
- 카드: 폭 400px, 흰 배경, 1px border `#E5E7EB`, border-radius 8px
- 카드 내부 패딩: 32px
- 카드 중앙 정렬 (수직·수평)
- 버튼: 각 provider 브랜드 컬러(또는 단색 outline) + 좌측 provider 로고 아이콘
- 버튼 사이 간격: 12px
- **로그인 실패 시**: 카드 하단에 에러 메시지 (예: "백오피스 접근 권한이 없습니다.")
#### OAuth 흐름
 
1. 운영자가 [Google/Kakao/Naver로 로그인] 클릭
2. 각 provider OAuth 팝업 → idToken 획득
3. `POST /api/v1/admin/auth/login` 호출 (`{ providerId, idToken }`)
4. role=ADMIN인 경우 토큰 발급 → 대시보드로 이동
5. role=USER인 경우 401/403 응답 → 에러 메시지 표시

```
┌──────────────────────────────────────┐
│  유저를 정지 처리하시겠습니까?    ✕  │  ← Title (16px bold)
├──────────────────────────────────────┤
│                                      │
│  유저 nickname (handle)을(를)        │  ← Description (14px)
│  SUSPENDED 상태로 변경합니다.        │
│                                      │
├──────────────────────────────────────┤
│              [취소]  [확인]          │  ← Footer
└──────────────────────────────────────┘
```
 
- shadcn/ui `Dialog` 컴포넌트 기반
- 폭 400px (단, 스테이징 promote 모달은 480px)
- [확인] 버튼 variant 정책:
| 액션 | Variant | 색상 | 비고 |
|---|---|---|---|
| 유저 status → SUSPENDED | default | `#7738EE` | |
| 유저 status → ACTIVE | default | `#7738EE` | |
| **유저 status → DELETED** | **destructive** | 빨강 | |
| 유저 role → USER | default | `#7738EE` | |
| **유저 role → ADMIN** (권한 상승) | **destructive** | 빨강 | |
| 신고 isResolved 토글 | default | `#7738EE` | |
| **스테이징 라벨 PENDING → COMPLETED** (최고 위험) | **destructive** | 빨강 | **라벨명 입력 확인 방식 (예외 처리, 6-7-3 참조)** |
| **공지 DELETE** | **destructive** | 빨강 | |
 
> **예외 처리 — 스테이징 promote**: 명세상 가장 위험한 액션이므로 단순 confirm이 아닌 **라벨명 입력 strict match**를 도입. 운영자가 라벨명을 정확히 타이핑해야 [반영] 버튼이 활성화됨. 상세 사양은 6-7-3 참조.
 
### 5-4. 인라인 편집 패턴 (화면별 차등)
 
| 화면 | 패턴 | 동작 |
|---|---|---|
| **Prod 학습 컨텐츠** (챕터/유닛/레슨/문제) | **B 패턴** | [편집] 버튼 → 편집 모드 → [저장]/[취소] |
| **스테이징 라벨 상세** | **A 패턴** | 항상 편집 가능, 변경 필드만 [저장] 활성화 |
| **공지 수정** | **B 패턴** | [편집] 버튼 → 편집 모드 → [저장]/[취소] |
| **유저 status/role** | 별도 | 드롭다운 선택 → confirm 모달 → API 호출 |
 
#### A 패턴 동작 상세 (스테이징 라벨 상세)
 
| 항목 | 동작 |
|---|---|
| **편집 트리거** | 별도 트리거 없음. 항상 편집 가능 |
| **변경 감지** | 초기 로드 값과 비교. 1개 이상 필드 변경 시 [저장] 버튼 활성화 |
| **변경 필드 시각 표시** | 변경된 input 좌측 4px 보더 `#7738EE` (`border-l-4`) |
| **저장 단위** | 항목 단위 ([저장] 버튼 1개로 변경된 필드만 다중 PATCH) |
| **부분 실패** | 성공 PATCH는 baseline 갱신, 실패 PATCH는 입력값/시각 표시 유지 |
| **항목 전환 시** | 자유 이동, 입력값 메모리 보존, 좌측 ●로 미저장 표시 |
| **페이지 이탈 시** | `beforeunload` 또는 라우터 confirm |
 
### 5-5. 토스트(Toast) 메시지 정책
 
- 액션 성공/실패 시 화면 우상단에 토스트 표시
- 성공: 초록 (예: "공지가 게시되었습니다.")
- 실패: 빨강 (예: "처리에 실패했습니다. 다시 시도해주세요.")
- 자동 dismiss: 3초

### 6-1. 대시보드 `/`

```
  (PENDING → COMPLETED 시 prod로 INSERT 실행, 가장 위험한 액션)
```

### 7-8. 공지

## 8. 도메인 데이터 사전
 
### 8-1. 유저
 
| 항목 | 값 |
|---|---|
| `status` | `ACTIVE`(활성), `SUSPENDED`(정지), `DELETED`(삭제됨) |
| `role` | `ADMIN`(관리자), `USER`(일반) |
| `profileImgNumber` | 정수 (사용자 앱과 동일한 프리셋 매핑 사용) |
 
### 8-2. 신고
 
| 항목 | 값 |
|---|---|
| `reportType` | `TYPO_ERROR`(오타), `CONTENT_ERROR`(내용 오류), `ANSWER_ERROR`(정답 오류), `OTHER_ERROR`(기타) |
| `isResolved` | `false`(미해결), `true`(해결됨) |
 
### 8-3. 학습 컨텐츠
 
| 항목 | 값 |
|---|---|
| `problemType` | `OBJECTIVE`(객관식, 4지선다 + 정답 1개), `SUBJECTIVE`(주관식, 정답 여러 개 허용) |
| 계층 구조 | Chapter → Unit → Lesson → Problem → Option(객관식) / Answer(주관식) |
 
### 8-4. 스테이징
 
| 항목 | 값 |
|---|---|
| `label` | `YYYY-MM-DD-update` 포맷. 한 라벨 = 한 사이클 (한 유닛의 레슨 1개 + 문제 6개) |
| `status` | `PENDING`(검수 대기), `COMPLETED`(반영 완료) |
| 위험 동작 | `PENDING → COMPLETED` 전환 시 prod에 INSERT 실행 (되돌리기 어려움) |
 
### 8-5. 공지
 
| 항목 | 값 |
|---|---|
| `status` | `DRAFT`(초안), `PUBLISHED`(게시중), `ARCHIVED`(보관됨) |
| `pinned` | boolean (상단 고정 여부) |
| 상태 전이 | DRAFT → PUBLISHED → ARCHIVED (단방향 진행, 역방향 불가) |
