# Gravit 백오피스 — Claude Design 프롬프트 템플릿
 
> **이 문서의 역할**: Claude Design과 대화할 때 **그대로 복사·붙여넣기**할 수 있는 프롬프트 템플릿 모음입니다.
>
> Claude Design 공식 가이드의 6가지 팁을 직접 반영했습니다:
> 1. 간단하게 시작 → 복잡성 계층화
> 2. 구체적 피드백
> 3. 디자인 시스템 컴포넌트 이름으로 호출
> 4. 반응형 정책 조기 명시
> 5. 변형 요청
> 6. Claude에게 검토 요청
 
---
 
## 사용 흐름
 
```
Step 1. 초기 컨텍스트 설정 (1회)
   ↓
Step 2. 화면별 첫 디자인 생성 (16번 반복, 한 번에 한 화면)
   ↓
Step 3. 화면별 변형 탐색 (방향이 불확실할 때)
   ↓
Step 4. 화면별 세부 조정 (구체적 피드백)
   ↓
Step 5. 디자인 시스템 일관성 검토
   ↓
Step 6. 접근성 / 사용성 검토
   ↓
Step 7. 인터랙티브 프로토타입 변환
```
 
---
 
## Step 1. 초기 컨텍스트 설정 프롬프트
 
> Claude Design과의 첫 대화 또는 디자인 시스템 설정 시 1회 사용.
 
```
Gravit 백오피스 디자인 작업을 시작합니다. 먼저 프로젝트 컨텍스트와 디자인 시스템을 설정합니다.
 
## 프로젝트 개요
- 서비스: CS(컴퓨터 과학) 학습 서비스의 내부 운영자용 백오피스
- 사용자: 내부 직원 수십 명
- 사용 환경: 사무실 데스크탑
 
## 디바이스 / 반응형 정책 (중요)
이 프로젝트는 **데스크탑 전용**입니다. 모바일·태블릿 변형이 필요하지 않으며, 반응형 break point도 없습니다. 모든 디자인은 1280px 이상의 단일 데스크탑 폭만 고려해주세요.
 
## 디자인 톤
- 미니멀 & 정보 중심 (Linear, Vercel Dashboard 계열)
- 라이트 모드 전용
- shadcn/ui (Radix + Tailwind) 기반
- 한국어 UI
 
## 디자인 시스템
첨부한 두 문서를 기반으로 디자인 시스템을 구축해주세요.
- DS-00_overview.md (프로젝트 개요)
- DS-01_design_system.md (컬러 토큰, 타이포 토큰, 컴포넌트 명명 규칙)
 
## 요청
위 내용을 기반으로 디자인 시스템을 등록하고, 준비가 되면 알려주세요. 이후 화면별로 디자인을 요청드리겠습니다.
```

## Step 3. 변형 탐색 프롬프트
 
> 디자인 방향이 확실하지 않을 때 사용. Claude Design은 한 번에 여러 변형을 만들 수 있습니다.
 
### 3-1. 변형 2~3개 동시 요청
 
```
{화면 ID} 화면의 {특정 영역, 예: 헤더 / 좌측 리스트 / Stat Card}에 대해 2~3개의 변형을 보여주세요. 각 변형의 특징을 설명해주시고, 어떤 변형이 어떤 사용 시나리오에 더 적합한지 함께 알려주세요.
```
 
### 3-2. 특정 변형 비교
 
```
대시보드의 Stat Card 디자인에 대해 다음 두 가지 변형을 비교해주세요.
 
A. 현재 디자인 (라벨 위, 큰 숫자 가운데, 링크 아래)
B. 큰 숫자를 좌측, 라벨과 링크를 우측에 배치한 가로형 레이아웃
 
각각의 정보 가독성, 시각적 균형, 정보 밀도 측면에서의 장단점을 알려주세요.
```

## Step 4. 세부 조정 프롬프트 (구체적 피드백)
 
> "이게 좀 이상해" 같은 모호한 피드백 대신 구체적으로.
 
### 4-1. 간격 조정
 
```
{화면 ID}의 {요소}에서 다음 간격을 조정해주세요.
- 폼 필드 간 세로 간격을 16px → 20px로 늘려주세요.
- 라벨과 입력 필드 사이 간격을 8px로 통일해주세요.
- 카드 사이 간격(card-gap)을 16px로 유지해주세요.
```
 
### 4-2. 정렬 조정
 
```
{화면 ID}의 데이터 테이블에서:
- "참여 인원" 컬럼의 숫자는 우측 정렬로 변경해주세요.
- "진행률" 컬럼의 Progress Bar와 % 텍스트는 좌측 정렬로 통일해주세요.
- 컬럼 헤더는 모두 좌측 정렬을 유지해주세요.
```
 
### 4-3. 컬러 조정
 
```
{화면 ID}의 {요소}에서:
- 활성 메뉴의 배경색을 디자인 시스템의 primary-bg-subtle(Primary 8% opacity)로 변경해주세요.
- 텍스트 색상은 Primary(#7738EE)로 강조해주세요.
- 비활성 메뉴의 텍스트는 text-secondary(#525252)로 유지해주세요.
```
 
### 4-4. 타이포 조정
 
```
{화면 ID}에서:
- 페이지 타이틀은 디자인 시스템의 h1 토큰(24px bold)을 적용해주세요.
- 섹션 타이틀은 h3 토큰(16px semibold)을 적용해주세요.
- 메타 정보(날짜, 카운트 등)는 caption 토큰(12px regular) + text-muted 색상으로 통일해주세요.
```
 
### 4-5. 컴포넌트 교체
 
```
{화면 ID}의 {요소}를 디자인 시스템의 {컴포넌트명}으로 교체해주세요.
 
예시:
- 현재 사용 중인 커스텀 버튼을 Destructive Button으로 교체
- 현재 사용 중인 커스텀 카드를 Info Card 패턴으로 교체
- 현재 사용 중인 커스텀 모달을 Confirm Modal 패턴으로 교체
```

## Step 5. 디자인 시스템 일관성 검토 프롬프트
 
> 16개 화면이 어느 정도 만들어진 후 사용. Claude를 디자인 협력자로 활용.
 
```
지금까지 만든 모든 화면을 디자인 시스템 일관성 측면에서 검토해주세요.
 
다음 항목들을 점검해주세요.
1. 컬러 토큰을 일관되게 사용하고 있는지 (특히 Primary 컬러는 액션과 강조에만 사용되어야 함)
2. 타이포 토큰이 일관되게 적용되었는지 (display/h1/h2/h3/body/caption)
3. 간격 토큰이 일관되게 적용되었는지 (card padding 24px, content padding 32px 등)
4. 동일한 패턴(예: List Page, Detail Page)이 화면 간 일관되게 적용되었는지
5. Status Badge의 variant 사용이 일관된지
 
발견된 불일치를 모두 알려주시고, 각각의 수정 방안을 제시해주세요.
```

## Step 6. 접근성 / 사용성 검토 프롬프트
 
> 디자인 1차 완료 후 사용. 디자이너 검수 전 사전 점검 용도.
 
### 6-1. 접근성 검토
 
```
{화면 ID} 화면의 접근성을 검토해주세요.
 
다음을 점검해주시고 개선안을 제안해주세요.
1. 텍스트와 배경의 명암비 (WCAG AA 기준 4.5:1 이상)
2. 인터랙티브 요소(버튼, 링크, input)의 클릭 영역 크기 (최소 44×44px 권장)
3. 키보드 네비게이션 가능성 (Tab 순서, 포커스 표시)
4. 폼 라벨과 input의 명확한 연결
5. 위험 액션(destructive)의 시각적 차별화 충분성
6. 한국어 텍스트 가독성 (글자 크기, 행간)
```
 
### 6-2. 정보 계층 구조 검토
 
```
{화면 ID} 화면의 정보 계층 구조(visual hierarchy)를 검토해주세요.
 
운영자가 이 화면에 진입했을 때 자연스럽게 시선이 이동하는 순서가 작업 흐름과 일치하는지 확인해주시고, 부자연스럽거나 개선이 필요한 부분을 알려주세요.
 
특히 다음 화면들은 작업 흐름이 중요합니다.
- USER_DETAIL: 운영자가 유저의 권한·상태를 변경하는 흐름
- REPORT_DETAIL: 운영자가 신고를 검토하고 처리하는 흐름  
- STAGING_DETAIL: 운영자가 7개 항목을 순차 검수하고 promote하는 흐름
```
 
### 6-3. 일반 사용성 검토
 
```
{화면 ID} 화면의 사용성을 검토해주세요.
 
다음 관점에서 분석해주세요.
1. 첫 방문자가 화면의 목적을 이해하는 데 걸리는 시간
2. 자주 사용되는 액션이 시각적으로 두드러지는지
3. 위험 액션이 실수로 클릭되기 쉬운 위치에 있지 않은지
4. 빈 상태 / 에러 상태에서 사용자가 다음 단계를 명확히 알 수 있는지
5. 정보 밀도가 너무 높거나 낮지 않은지
 
발견된 문제와 개선안을 함께 알려주세요.
```

## Step 7. 인터랙티브 프로토타입 변환 프롬프트
 
> 정적 디자인이 확정된 후 인터랙티브 프로토타입으로 변환할 때 사용.
 
```
지금까지 만든 화면을 인터랙티브 프로토타입으로 변환해주세요. 다음 상호작용 명세를 따라주세요.
 
[여기에 DS-03_interactions.md의 해당 화면 섹션 또는 전체 붙여넣기]
 
특히 다음 핵심 상호작용을 정확히 구현해주세요.
1. 사이드바 네비게이션 (메뉴 클릭 → 해당 화면 이동)
2. 데이터 테이블 행 클릭 → 상세 페이지 이동
3. Confirm Modal 흐름 (위험 액션은 destructive variant)
4. 토스트 메시지 (성공/실패, 3초 자동 dismiss)
5. STAGING_DETAIL의 Strict Match Modal (라벨명 정확히 일치해야 [반영] 활성화)
```

```
이 모달을 Confirm Modal 패턴으로 변경해주세요.
```
 
### 검토 요청용
 
```
이 디자인의 접근성을 검토해주세요.
이 디자인의 디자인 시스템 일관성을 검토해주세요.
이 화면의 정보 계층이 운영 작업 흐름에 적합한지 검토해주세요.
```

## 사용 시 주의사항
 
### ❌ 피해야 할 프롬프트 패턴
 
```
"좀 더 예쁘게 만들어주세요"
   → "예쁘다"의 기준이 모호. 구체적 변경점 명시 필요.
 
"여기가 이상해 보여요"
   → 어디가 어떻게 이상한지 명확하지 않음.
 
"전체적으로 조정해주세요"
   → 무엇을 조정할지 알 수 없음.
 
"모바일에서도 잘 보이게 해주세요"
   → 데스크탑 전용 프로젝트. 반응형 변형 요청 금지.
```
 
### ✅ 권장 프롬프트 패턴
 
```
"폼 필드 사이 간격을 16px → 20px로 늘려주세요" (구체적 수치)
"이 버튼을 Destructive Button variant로 변경해주세요" (디자인 시스템 컴포넌트명)
"두 가지 변형으로 보여주세요: A안과 B안" (변형 요청)
"이 화면의 접근성을 검토해주세요" (검토 요청)
```
